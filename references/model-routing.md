# 模型路由协议

在派发任何 Dev、Integrator、Reviewer 或测试子会话前读取本文件。这里使用相对能力档位，不维护具体模型版本表；产品模型升级后，运行时依据当前任务模型重新解析即可。

## 角色与默认路由

- `CURRENT`：启动当前主任务时用户选择或系统分配的模型。
- `LOWER`：运行时可用、能力比 `CURRENT` 低一级且仍适合生产开发的模型；若有多个候选，选择最接近 `CURRENT` 的一级。
- 未明确指定时，所有子会话的推理强度默认使用 `medium`。Integrator、Reviewer 和本协议列出的复杂或高风险任务固定使用 `high`；若 `high` 不可用，暂停受影响节点并请求用户确认，不自动回退。只有用户明确点名具体推理档位才构成覆盖；“优先速度”“尽快”等笼统偏好不能突破角色或风险下限。
- 需求讨论、OpenSpec 完善和 Wayfinder 规划不属于本 Skill。若希望“需求确认和 Review 用最好模型、开发优先速度”，应在启动主任务时选择最好模型。
- 主任务始终沿用 `CURRENT`，Skill 不静默切换主任务模型。
- 子任务根据复杂度在 `LOWER` 与 `CURRENT` 之间选择，不自动使用强于 `CURRENT` 的模型。
- Integrator 使用 `CURRENT` + `high`，负责跨任务语义、冲突和整体验证。
- 独立 Reviewer 使用 `CURRENT` + `high`，并保持只读。独立性要求 fresh 会话和独立上下文，不要求实际模型必须不同；Dev 自检不能代替它。

## 解析顺序

1. 用户明确指定的模型、具体推理档位或“所有子任务只用一个模型”优先；笼统的速度偏好不视为推理档位覆盖。即使 Reviewer 因此使用与 Dev 相同的模型，也必须是 fresh 独立会话。
2. 否则把主任务实际模型记为 `CURRENT`，再从运行时暴露的子会话选项、官方能力标签或元数据中解析 `LOWER`。
3. 无法可靠确认低一级模型、运行时不支持覆盖或只提供一个模型时，子任务继承 `CURRENT`；不得猜测或为了速度跨越多个能力档位。
4. 不猜测、不拼接不可用的模型 ID；fallback 不阻塞交付，除非用户把精确模型指定为硬要求。

每次派发记录角色层级、实际模型（可见时）、推理强度、选择原因、是否偏离默认中等档和 fallback。单次任务可在提示语中覆盖本策略，不需要修改 Skill。

## 复杂度判断

仅当以下条件全部成立时，Dev 或测试执行子任务可使用 `LOWER` + `medium`：

- 需求、契约和验收标准已冻结，任务边界窄且可独立验证；
- 修改集中在单一模块，不涉及共享 API/DTO/schema、数据库迁移或跨模块语义；
- 不涉及资金、支付、账务、权限、认证、安全、隐私、并发一致性或不可逆副作用；
- 实现路径清楚，失败可安全回滚，focused tests 能直接判断正确性；
- 不存在 P0/P1、重复失败或需要复杂故障归因的情况。

任一条件不成立，使用 `CURRENT`。测试设计、复杂失败诊断、真实 HTTP 业务验收也使用 `CURRENT`；只有确定性命令执行、日志采集和低风险机械检查可使用 `LOWER`。

以下任务使用 `CURRENT` + `high`：

- 资金、支付、账务、权限、认证、安全、隐私；
- 数据库迁移、并发一致性、共享 API/DTO/schema/核心契约；
- 跨模块复杂调用链、难回滚或不可逆副作用；
- 已确认 P0/P1，或 Reviewer 明确要求高能力修复；
- 同一失败经过两轮有效修复仍重复出现；
- 实现中重新出现会改变业务语义、范围或验收标准的歧义。

最后一种情况还必须按主 Skill 门禁暂停受影响节点并请求确认，提高推理强度不能替代需求决策。P2/P3 默认按复杂度重新路由；若涉及上述高风险面或重复失败，同样使用 `CURRENT`。

“原 writer”表示责任归属，不要求复用原子会话。切换 writer 或模型档位时，必须依次停止旧 writer、固定并核对 checkpoint/workspace、完成交接、退休并释放 lease，再启动只读 fresh replacement；核验通过后才记录新的 lease owner 并授权写入。任一步失败即 `BLOCKED`，不得保留两个 writer 并行修改，也不得重新激活退休会话。

如果 `CURRENT` 对任务明显不足，停止受影响节点并建议用户为主任务选择更强模型；不得由 Skill 静默突破 `CURRENT` 上限。

## 速度与版本维护

- 速度优化优先通过合格任务使用 `LOWER`、安全并行和减少重复验证实现。笼统的速度偏好不降低 `medium` 默认值或 `high` 角色/风险下限；只有用户明确点名具体推理档位时才覆盖。Integrator、最终 Review 和复杂或高风险修复始终使用 `CURRENT`。
- 不自动开启可能改变费用、额度或服务等级的 Fast/priority 模式；只有用户明确要求并授权时使用。
- 具体模型名称变化不需要更新本 Skill。只有官方能力分层、子会话覆盖接口或推理参数发生语义变化，或 fresh 推演证明路由失效时，才调整本协议。

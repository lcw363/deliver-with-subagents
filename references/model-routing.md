# 模型路由协议

在派发任何 Dev、Integrator、Reviewer 或测试子会话前读取本文件。这里使用能力角色，不维护具体模型版本表；产品模型升级后，运行时重新解析当前可用模型即可。

## 角色与默认路由

- `FLAGSHIP`：当前可用模型中，复杂推理、代码理解和审查能力最强的一档。
- `BALANCED`：当前可用模型中，兼顾开发质量、延迟和成本的较快一档。
- 需求讨论、OpenSpec 完善和 Wayfinder 规划不属于本 Skill。建议在进入交付前，由用户选择 `FLAGSHIP` 与最高实用推理强度完成确认。
- 主会话沿用用户已选择的模型，不承诺在会话中静默切换；它只负责调度、压缩状态和最终判断。
- 常规 Dev 使用 `BALANCED` + 中等推理；纯机械测试执行或低风险小改可使用较低至中等推理。
- Integrator 使用 `FLAGSHIP` + 高推理，负责跨任务语义、冲突和整体验证。
- 独立 Reviewer 默认使用 `FLAGSHIP` + 最高实用推理，并保持只读。独立性要求 fresh 会话和独立上下文，不要求实际模型必须不同；Dev 自检不能代替它。
- 普通修复交回责任 writer，并沿用其角色默认路由：Dev 为 `BALANCED` + 中等推理，Integrator 为 `FLAGSHIP` + 高推理。命中升级条件时，从最新 fixed point 启动 fresh `FLAGSHIP` writer，并先完成 writer lease 交接。

## 解析顺序

1. 用户明确指定的模型、推理强度或“所有子任务只用一个模型”优先；即使 Reviewer 因此使用与 Dev 相同的模型，也必须是 fresh 独立会话。
2. 否则从当前会话实际暴露的子会话模型选项、官方能力标签或运行时元数据中解析 `FLAGSHIP` 与 `BALANCED`。
3. 无法可靠区分、没有覆盖能力或只提供一个模型时，继承父会话模型，并使用该模型支持的最接近推理强度。
4. 不猜测、不拼接不可用的模型 ID；fallback 不阻塞交付，除非用户把精确模型指定为硬要求。

每次派发记录角色层级、实际模型（可见时）、推理强度、选择原因和 fallback。单次任务可在提示语中覆盖本策略，不需要修改 Skill。

## 自动升级

以下任务的实现或修复直接使用 `FLAGSHIP` + 高或更高推理：

- 资金、支付、账务、权限、认证、安全、隐私；
- 数据库迁移、并发一致性、共享 API/DTO/schema/核心契约；
- 跨模块复杂调用链、难回滚或不可逆副作用；
- 已确认 P0/P1，或 Reviewer 明确要求高能力修复；
- 同一失败经过两轮有效修复仍重复出现；
- 实现中重新出现会改变业务语义、范围或验收标准的歧义。

最后一种情况还必须按主 Skill 门禁暂停受影响节点并请求确认，模型升级不能替代需求决策。P2/P3 默认由原 writer 修复；若涉及上述高风险面或重复失败，同样升级。

“原 writer”表示责任归属，不要求复用原子会话。升级时必须依次停止旧 writer、固定并核对 checkpoint/workspace、记录新的 lease owner，再启动 fresh `FLAGSHIP` writer；任一步失败即 `BLOCKED`，不得保留两个 writer 并行修改。

## 速度与版本维护

- 把速度优化放在常规 Dev、机械测试和可安全并行的只读任务；Integrator、最终 Review 和高风险修复不降级。
- 不自动开启可能改变费用、额度或服务等级的 Fast/priority 模式；只有用户明确要求并授权时使用。
- 具体模型名称变化不需要更新本 Skill。只有官方能力分层、子会话覆盖接口或推理参数发生语义变化，或 fresh 推演证明路由失效时，才调整本协议。

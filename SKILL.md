# 子会话交付闭环

以主会话为调度者，以一个 Dev 子会话为唯一代码写入者。把实现、日志和测试细节留在子会话，只向主会话返回可核验摘要。

## 使用边界

本 Skill 只用于需求已经确认后的开发交付阶段，包括实现、测试、独立 Review、修复和复查；不负责需求发现、方案讨论或范围澄清。

开始开发前，建议先使用 OpenSpec 把需求、验收标准和任务沉淀为可执行规格；大型、跨阶段或需要多个会话推进的工作，建议先使用 Wayfinder 明确路线、依赖和阶段边界。仍有关键需求未决时，应先返回需求与规划流程，不把猜测带入生产代码。

## 1. 锁定输入和边界

1. 读取适用的 `AGENTS.md`、项目规则和用户指令。
2. 按以下顺序确定需求真值：用户当前指令、明确指定的 OpenSpec/PRD/ticket、仓库现有行为与文档。
3. OpenSpec 存在时，读取对应 `proposal.md`、`design.md`、`tasks.md` 和增量 specs，只实现指定 task 或 phase。Wayfinder 仍有未决路线时，先报告规划未完成，不把猜测写进生产代码。
4. OpenSpec 工具可用时，先验证对应 change；结构或语义校验失败时先修正规格，不进入实现。
5. 记录目标、验收标准、不做内容、受影响范围、测试策略、收尾时限和发布边界。保留工作区已有无关改动。
6. 对低风险、可逆且不改变业务语义的缺口，结合现有模式继续处理并记录假设。若仍无法确定，且选择会影响业务、数据、权限、安全、接口契约或范围，暂停相关修改并请求用户确认。
7. 除非用户明确要求，不执行 commit、push、PR、部署或 OpenSpec archive。

完成标准：主会话得到一份足以交给子会话执行、无需携带整段历史的精简任务简报。

## 2. 启动唯一 Dev 子会话

启动一个 Dev 子会话并传入精简任务简报。任何时刻只允许这个子会话修改代码；其他子会话保持只读。

在 Dev 提示中明确要求：

- 使用 `$implement` 推进指定工作，但把 Review 和发布留给主会话；完成实现与验证后返回，不自行 commit 或 push。
- 仅在行为可测试且公共测试缝清晰时使用 `$tdd`。把明确的验收标准、公开接口和现有测试模式视为已约定缝；只有测试缝选择会实质改变设计时才请求确认。
- Bug 修复优先先写能复现问题的回归测试，再做最小修复。
- 第一版完成后使用 `$code-simplifier` 简化本次改动并补充必要注释，保持行为、API、数据和错误契约不变。
- Java/Spring/MyBatis 项目在相关 Skill 可用时使用 `$asana-openspec-java-workflow:java-test-strategy`；构建失败时使用 `$asana-openspec-java-workflow:java-build-fix` 做最小修复。其他技术栈遵循项目已有测试工具和命令。
- 把完整命令输出、实现过程和测试日志留在子会话，只返回改动摘要、关键文件、测试命令及结果、未完成项、阻塞和残余风险。

主会话等待 Dev 完成，不重复执行完整实现过程。

完成标准：第一版功能已实现并完成最窄有效验证；未完成内容和阻塞已显式列出。

## 3. 执行测试闭环

让 Dev 子会话按以下层级验证：

1. 开发中运行单个或最相关测试，保持快速反馈。
2. 简化或修复后重新运行原失败测试和受影响模块测试。
3. 最终运行适用的单元测试、集成测试、类型检查、编译和构建；时间允许时运行一次完整测试套件。
4. 新增或修改 HTTP API 时，除自动化测试外，启动实际本地服务或使用明确授权的测试环境，以具体、可执行、可回滚的参数调用真实路由。验证 HTTP 状态、响应字段、业务结果和必要的数据副作用。
5. 通过环境变量提供测试 Token，脱敏请求与响应中的凭据和个人信息；64 位 ID 在 JSON 中使用字符串。未经明确授权，不访问生产环境或使用不可回滚的生产数据。
6. 无法启动服务、缺少凭据或没有有效 fixture 时标记 `BLOCKED`，列出缺口；不得用 MockMvc、单测或 schema 检查冒充真实接口验收。
7. OpenSpec 存在时，在实现后重新验证 change；只把已有实现与验证证据的 tasks 标为完成，保留真实接口验收等未完成门禁。

实际接口验证摘要至少包含：接口、环境、脱敏后的具体参数、预期结果、实际 HTTP 状态、响应摘要、数据副作用、清理或回滚结果，以及 `PASS`、`FAIL` 或 `BLOCKED`。

完成标准：每项验收标准都有自动化或实际接口证据，或者有明确的未验证原因和风险。

## 4. 独立 Review

不要在 Dev 子会话内执行最终 Review。

- 已提交分支且用户给出 fixed point 时，由主会话显式使用 `$code-review`；该 Skill 自行运行 Standards 与 Spec 子代理，不要再包一层 Review 子会话。
- 未提交工作区改动时，优先使用专用的“Review uncommitted changes”；若当前界面不可用，启动一个只读 Review 子会话，检查 staged、unstaged 和本任务相关的 untracked files，并对照需求、项目标准和直接影响链路。
- Review 只返回可执行问题：等级、文件位置、证据、影响、最小修复建议、测试缺口和残余风险。原始日志和中间分析留在 Review 子会话。

严重度统一为：

- P0：可能造成灾难性数据、安全或全局可用性事故。
- P1：阻断核心需求、主流程或发布的严重缺陷。
- P2：重要场景中的真实正确性、兼容性或可靠性问题。
- P3：低影响但可复现、应在当前范围修复的局部缺陷。

纯风格偏好和没有具体失败路径的建议不列为 P3。

完成标准：主会话收到按严重度排序、带证据的压缩 Review 结果；没有问题时明确写明审查范围和“未发现已知且可执行的 P0-P3 问题”。

## 5. 修复与复查

1. 主会话只判断 Review 结论是否成立，不重复完整 Review。
2. 把确认成立的问题摘要交回原 Dev 子会话；Dev 做最小修复、补回归测试并重新执行原失败请求及受影响测试。
3. 区分失败来源：实现 Bug 回 Dev；需求偏差先更新并确认 OpenSpec；fixture 或环境问题修复测试条件；无关历史失败单独记录，不扩入当前任务。
4. 修复后重新执行独立 Review。
5. 收尾时限以用户指令为准；用户未指定时，从第一版实现完成开始默认计时 60 分钟。用户可指定其他时长，也可明确要求不限时持续到完成。
6. 达到适用时限后立即停止继续修改，并汇报未完成项、未验证项、阻塞和残余风险。不限时不代表可以跳过用户确认、权限、安全或环境边界；遇到这些阻塞时仍须暂停。

完成标准：在已审查和验证范围内没有已知且可执行的 P0-P3 问题，或者已达到适用时限并完整列出剩余工作。

## 6. 主会话最终输出

只汇总：

- 需求和 OpenSpec/task 完成情况；
- 主要改动与关键文件；
- 简化和关键注释情况；
- Review 发现、修复和复查结果；
- 自动化测试与实际接口参数测试证据；
- 当前已知 P0-P3 问题；
- 未完成事项、待确认问题、未验证项和残余风险；
- commit、push、部署和 archive 状态。

没有相关内容时写“无”。区分代码完成、自动化测试通过、真实接口验收通过和上线完成，不用其中一项替代另一项。
